# Implementation Plan: Incremental Ingestion with Watermark and Manifest Tracking

**Branch**: `008-incremental-ingestion` | **Date**: 2026-02-25 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/008-incremental-ingestion/spec.md`

## Summary

Add a DuckDB-backed file manifest layer to the ingestion pipeline so that
subsequent pipeline runs skip files already ingested (same path + same SHA-256
hash), detect modified files (same path, different hash), and support
`--full-refresh` for forced re-ingestion. A watermark table accelerates
startup by pre-filtering files via file-system mtime before computing hashes.
The existing `validator.py`, `dedup.py`, and `loader.py` are unchanged; all
new behavior is encapsulated in a new `src/ingestion/manifest.py` module with
targeted surgical changes to `pipeline.py`, `models.py`, and
`ingest_transactions.py`.

## Technical Context

## Architecture Overview

### Manifest Layer as a Wrapper

The manifest layer sits between file *discovery* and file *processing*. It does
not replace any existing stage; it adds a decision gate:

```
discover_parquet_files()
        │
        ▼
filter_by_mtime_watermark()     ← new: cheap pre-filter, skips files below watermark
        │
        ▼
compute_file_hash()             ← new: SHA-256 of raw bytes for candidate files
        │
        ▼
manifest.get_pending_files()    ← new: returns only files not in manifest as success+same-hash
        │
   ┌────┴──────────────────────────┐
   │ for each pending file:        │
   │   manifest.upsert_pending()   │ ← new: two-phase commit, phase 1
   │   _process_file()             │ ← existing, unchanged
   │   manifest.mark_success/fail()│ ← new: two-phase commit, phase 2
   └───────────────────────────────┘
        │
        ▼
manifest.update_watermark()     ← new: advance watermark to max mtime of successful files
```

The `--full-refresh` flag bypasses the mtime and hash checks: all files are
treated as pending (manifest entries are reset to `pending` in bulk before
the loop begins). The `manifest_metadata` watermark row is deleted (not set
to NULL, which would violate the `NOT NULL` constraint) so `get_watermark()`
returns `None` and all files pass the mtime pre-filter.

## New Module: `src/ingestion/manifest.py`

### `ManifestEntry` dataclass

```python
@dataclass
class ManifestEntry:
    file_path: str          # relative path from source_dir root (primary key)
    file_size_bytes: int
    file_hash: str          # SHA-256 hex digest
    file_mtime: float       # Unix timestamp (os.path.getmtime)
    run_id: str
    processed_at: str       # ISO-format UTC timestamp
    status: str             # 'pending' | 'success' | 'failed'
    error_message: str | None = None
```

### `compute_file_hash(path: Path) -> str`

Streams raw file bytes in 64 KB chunks through `hashlib.sha256`. Returns the
hex digest. Never parses Parquet — operates at byte level only (Clarification Q1).

```python
def compute_file_hash(path: Path, chunk_size: int = 65536) -> str:
    h = hashlib.sha256()
    with path.open("rb") as f:
        for chunk in iter(lambda: f.read(chunk_size), b""):
            h.update(chunk)
    return h.hexdigest()
```

### `ManifestStore` class

Owns all DuckDB interactions for the manifest. Initialized with a live
`duckdb.DuckDBPyConnection` so it shares the same connection (and therefore
the same transaction context) as the rest of the pipeline.

#### `create_tables() -> None`

Executes DDL for `file_manifest` and `manifest_metadata` if they do not exist
(FR-006). Safe to call on every run (idempotent `CREATE TABLE IF NOT EXISTS`).

#### `get_watermark(source_dir: str) -> float | None`

```sql
SELECT watermark_mtime FROM manifest_metadata WHERE source_dir = ?
```
Returns `None` if no watermark has been recorded yet (first run or after
`--full-refresh` deletes the row).

#### `delete_watermark(source_dir: str) -> None`

```sql
DELETE FROM manifest_metadata WHERE source_dir = ?
```
Called by `run_pipeline()` when `full_refresh=True`, before the file loop.
Deleting the row (rather than setting `watermark_mtime = NULL`, which would
violate the `NOT NULL` constraint) causes `get_watermark()` to return `None`
and forces all files to pass the mtime pre-filter (C-003 resolution).

#### `update_watermark(source_dir: str, mtime: float) -> None`

```sql
INSERT INTO manifest_metadata (source_dir, watermark_mtime, updated_at)
VALUES (?, ?, NOW())
ON CONFLICT (source_dir) DO UPDATE SET
    watermark_mtime = excluded.watermark_mtime,
    updated_at = excluded.updated_at
```

#### `get_manifest_entry(file_path: str) -> ManifestEntry | None`

```sql
SELECT * FROM file_manifest WHERE file_path = ?
```
Returns `None` if the file has never been ingested.

#### `upsert_pending(file_path: str, file_hash: str, file_size_bytes: int, file_mtime: float, run_id: str) -> None`

Two-phase commit phase 1 (FR-012, Clarification Q5). Writes the entry with
`status = 'pending'` and current `run_id` before ingestion begins.

```sql
INSERT INTO file_manifest (
    file_path, file_size_bytes, file_hash, file_mtime,
    run_id, processed_at, status, error_message
) VALUES (?, ?, ?, ?, ?, ?, 'pending', NULL)
ON CONFLICT (file_path) DO UPDATE SET
    file_size_bytes = excluded.file_size_bytes,
    file_hash       = excluded.file_hash,
    file_mtime      = excluded.file_mtime,
    run_id          = excluded.run_id,
    processed_at    = excluded.processed_at,
    status          = 'pending',
    error_message   = NULL
```

#### `mark_success(file_path: str) -> None`

Two-phase commit phase 2 — success path (FR-012).

```sql
UPDATE file_manifest
SET status = 'success', processed_at = NOW()
WHERE file_path = ?
```

#### `mark_failed(file_path: str, error: str) -> None`

Two-phase commit phase 2 — failure path (FR-012).

```sql
UPDATE file_manifest
SET status = 'failed', error_message = ?, processed_at = NOW()
WHERE file_path = ?
```

#### `reset_all_to_pending(run_id: str) -> None`

Used by `--full-refresh`. Bulk-updates all existing entries (Clarification Q3).

```sql
UPDATE file_manifest
SET status = 'pending', run_id = ?, error_message = NULL, processed_at = NOW()
```

#### `should_skip(file_path: str, file_hash: str) -> bool`

Returns `True` if the file is already in the manifest with `status='success'`
and the same hash (FR-003). Files with `status='failed'` or different hash
always return `False` (FR-004, FR-005).

## Changes to `src/ingestion/pipeline.py`

### New function: `_filter_files_by_manifest`

```python
def _filter_files_by_manifest(
    files: list[Path],
    manifest: ManifestStore,
    source_dir: Path,
    full_refresh: bool,
) -> tuple[list[tuple[Path, str]], int]:
    """
    Returns:
        - List of (file_path, file_hash) tuples to process
        - Number of files skipped (already in manifest as success + same hash)
    """
```

Logic:
1. Get watermark from manifest (skip if `full_refresh`).
2. For each file, compute relative path string.
3. If not `full_refresh` and `file.stat().st_mtime <= watermark`: skip without
   hashing (mtime pre-filter, FR-010).
4. Compute `file_hash = compute_file_hash(file)`.
5. If not `full_refresh` and `manifest.should_skip(rel_path, file_hash)`: skip.
6. Append to pending list.

### Changes to `run_pipeline`

New signature:

```python
def run_pipeline(
    *,
    source_dir: str | Path = DEFAULT_SOURCE_DIR,
    db_path: str | Path = DEFAULT_DB_PATH,
    full_refresh: bool = False,          # NEW
) -> RunResult:
```

New logic inside `run_pipeline`:

1. After `create_tables(conn)`, call `manifest.create_tables()`.
2. If `full_refresh`: call `manifest.reset_all_to_pending(run_id)`.
3. Replace the bare `files` loop with `_filter_files_by_manifest()` to get
   `(file_path, file_hash)` pairs and a `files_skipped` count.
4. For each `(file_path, file_hash)` in the pending list:
   a. Build `ManifestEntry` with `status='pending'`.
   b. Call `manifest.upsert_pending(entry)`.
   c. Call `_process_file(conn, file_path, run_id, existing_ids)`.
   d. On success (no error in `FileResult`): call `manifest.mark_success(rel_path)`.
   e. On failure (FileResult.error is not None): call `manifest.mark_failed(rel_path, error)`.
5. After all files: call `manifest.update_watermark(str(source_dir), max_mtime)`.
6. Extend `complete_run()` call with new manifest stats: `files_checked`,
   `files_skipped`, `files_ingested`, `files_failed`.

## Backward Compatibility

- On first run, `file_manifest` and `manifest_metadata` tables are created
  automatically (FR-006). No migration needed.
- Existing DuckDB databases without the manifest tables are handled
  transparently: `CREATE TABLE IF NOT EXISTS` ensures idempotency.
- `ingestion_runs` table gains four new nullable columns via `ALTER TABLE IF
  NOT EXISTS` or included in the DDL `IF NOT EXISTS` path — DuckDB will not
  error if the table already exists without those columns. A migration guard
  will `ALTER TABLE ingestion_runs ADD COLUMN IF NOT EXISTS ...` for existing
  DBs.
- `validator.py`, `dedup.py` are untouched. All existing Feature 002 tests
  continue to pass (SC-008).
- Default behavior without `--full-refresh` is incremental mode; no
  behavioral change for callers that do not pass the flag.

## SQL DDL Reference

### `file_manifest`

```sql
CREATE TABLE IF NOT EXISTS file_manifest (
    file_path        VARCHAR PRIMARY KEY,
    file_size_bytes  BIGINT  NOT NULL,
    file_hash        VARCHAR NOT NULL,          -- SHA-256 hex digest
    file_mtime       DOUBLE  NOT NULL,          -- Unix timestamp (float)
    run_id           VARCHAR NOT NULL,          -- FK → ingestion_runs.run_id
    processed_at     TIMESTAMPTZ NOT NULL,
    status           VARCHAR NOT NULL           -- 'pending' | 'success' | 'failed'
                     CHECK (status IN ('pending', 'success', 'failed')),
    error_message    VARCHAR                    -- NULL for success entries
);
```

### `manifest_metadata`

```sql
CREATE TABLE IF NOT EXISTS manifest_metadata (
    source_dir       VARCHAR PRIMARY KEY,
    watermark_mtime  DOUBLE  NOT NULL,          -- Unix timestamp of max processed mtime
    updated_at       TIMESTAMPTZ NOT NULL
);
```
