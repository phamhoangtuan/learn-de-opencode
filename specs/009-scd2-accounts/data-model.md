# Data Model: SCD Type 2 — Accounts Dimension

**Feature**: 009-scd2-accounts
**Date**: 2026-02-26
**Warehouse**: `data/warehouse/transactions.duckdb`

---

## Overview

This feature introduces two new objects (`dim_accounts_sk_seq` sequence and
`dim_accounts` table) and one new SQL transform (`mart__account_history` view).
All existing tables (`transactions`, `quarantine`, `ingestion_runs`,
`file_manifest`, `manifest_metadata`) are unchanged.

The `dim_accounts` table implements SCD Type 2 change tracking. Each row
represents a single version of an account's behavioral profile, derived from
the full transaction history. When attributes change between pipeline runs, the
previous row is expired (closed) and a new current row is inserted. Historical
rows are retained indefinitely.

---

## Entity Catalogue

---

### Sequence: `dim_accounts_sk_seq`

**Purpose**: Generates monotonically increasing surrogate key values for
`dim_accounts.account_sk`. Generated SQL-side via `nextval('dim_accounts_sk_seq')`
inside each INSERT statement, consistent with the existing `quarantine_id_seq`
pattern in `loader.py`.

```sql
CREATE SEQUENCE IF NOT EXISTS dim_accounts_sk_seq START 1;
```

**Notes**:
- The sequence starts at 1. Values are assigned in insertion order across all
  pipeline runs.
- The sequence is never reset or reused. Retired (expired) rows retain their
  original surrogate key permanently.
- `CREATE SEQUENCE IF NOT EXISTS` makes DDL idempotent — safe to call in
  `create_tables()` on every pipeline startup.

---

### Table: `dim_accounts`

**Purpose**: SCD Type 2 dimension table. Each row is a version of an account's
behavioral profile derived from the full transaction history up to the time of
the pipeline run that inserted it. The natural key is `account_id`; the
surrogate key is `account_sk`. Exactly one row per `account_id` has
`is_current = true` at any point in time.

**Primary key**: `account_sk` — sequence-generated `BIGINT`.
**Natural key**: `account_id` — the business identifier, not unique across all
rows (multiple versions of the same account will share this value).

```sql
CREATE SEQUENCE IF NOT EXISTS dim_accounts_sk_seq START 1;

CREATE TABLE IF NOT EXISTS dim_accounts (
    account_sk          BIGINT      PRIMARY KEY
                                    DEFAULT nextval('dim_accounts_sk_seq'),
    account_id          VARCHAR     NOT NULL,
    primary_currency    VARCHAR     NOT NULL,
    primary_category    VARCHAR     NOT NULL,
    transaction_count   BIGINT      NOT NULL,
    total_spend         DOUBLE      NOT NULL,
    first_seen          DATE        NOT NULL,
    last_seen           DATE        NOT NULL,
    row_hash            VARCHAR     NOT NULL,
    valid_from          TIMESTAMPTZ NOT NULL,
    valid_to            TIMESTAMPTZ NOT NULL,
    is_current          BOOLEAN     NOT NULL
);
```

### Column Descriptions

| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| `account_sk` | `BIGINT` | NOT NULL | Surrogate key. Auto-generated by `dim_accounts_sk_seq` at INSERT time. Unique across all rows including expired versions. Never reused. |
| `account_id` | `VARCHAR` | NOT NULL | Natural key. Business identifier for the account (e.g., `ACC-00042`). Not unique across all rows — multiple historical versions of the same account share this value. Must match the `account_id` pattern used in the `transactions` table. |
| `primary_currency` | `VARCHAR` | NOT NULL | The most frequently used currency in this account's full transaction history at the time this version was created (mode, alphabetically first on tie). Example: `"USD"`. |
| `primary_category` | `VARCHAR` | NOT NULL | The most frequently used spending category in this account's full transaction history at the time this version was created (mode, alphabetically first on tie). Example: `"Groceries"`. |
| `transaction_count` | `BIGINT` | NOT NULL | Total number of transactions for this account in the full transaction history at the time this version was created. |
| `total_spend` | `DOUBLE` | NOT NULL | Sum of `amount` across all transactions for this account in the full history at the time this version was created. |
| `first_seen` | `DATE` | NOT NULL | Earliest `transaction_date` in this account's full transaction history. |
| `last_seen` | `DATE` | NOT NULL | Latest `transaction_date` in this account's full transaction history. |
| `row_hash` | `VARCHAR` | NOT NULL | MD5 fingerprint of the six tracked attribute columns used for change detection. Computed as `md5(primary_currency \|\| '\|' \|\| primary_category \|\| '\|' \|\| transaction_count \|\| '\|' \|\| total_spend \|\| '\|' \|\| first_seen \|\| '\|' \|\| last_seen)` with all values COALESCE'd to `''`. A hash mismatch between the newly computed profile and the stored value triggers a new version. |
| `valid_from` | `TIMESTAMPTZ` | NOT NULL | UTC timestamp of the pipeline run that created this version row. For new accounts (no prior row), set to the run timestamp. For changed accounts (replacement row), set to the same run timestamp used to close the previous row. |
| `valid_to` | `TIMESTAMPTZ` | NOT NULL | UTC timestamp marking when this version became inactive. For current rows (`is_current = true`), set to the far-future sentinel `9999-12-31T23:59:59+00:00`. For expired rows, set to the run timestamp of the pipeline run that detected the attribute change. |
| `is_current` | `BOOLEAN` | NOT NULL | True if this row represents the current (latest) version of the account. Exactly one row per `account_id` must have `is_current = true` at all times. Expired rows have `is_current = false`. |

### Indexes

DuckDB does not require explicit index declarations for small-to-medium
analytical workloads. The primary key on `account_sk` provides an implicit
unique index. The following query patterns drive index considerations:

- **Change detection lookup** — `WHERE account_id = ? AND is_current = true`:
  the most frequent access pattern during the SCD merge. For datasets up to
  100,000 transactions, a full scan of `dim_accounts` (bounded by the number
  of unique accounts, not transactions) is acceptable per SC-005.
- **History queries** — `WHERE account_id = ? ORDER BY valid_from`: satisfied
  by a full scan of the small dimension table.

No additional indexes are required for current scale targets.

### View: `mart__account_history`

**Purpose**: Analysis-ready SQL view over `dim_accounts` that exposes the
complete version timeline for every account. Adds two computed columns —
`version_number` (chronological version counter per account) and
`version_duration_days` (how long each version was or has been active) — so
that consumers can build reports on account behavior evolution without
understanding SCD2 mechanics.

**Depends on**: `dim_accounts`

```sql
-- model: mart__account_history
-- depends_on: dim_accounts
CREATE OR REPLACE VIEW mart__account_history AS
SELECT
    account_sk,
    account_id,
    primary_currency,
    primary_category,
    transaction_count,
    total_spend,
    first_seen,
    last_seen,
    valid_from,
    valid_to,
    is_current,
    ROW_NUMBER() OVER (
        PARTITION BY account_id
        ORDER BY valid_from ASC
    )                                                        AS version_number,
    CASE
        WHEN is_current THEN
            CAST(CURRENT_DATE - CAST(valid_from AS DATE) AS BIGINT)
        ELSE
            CAST(CAST(valid_to AS DATE) - CAST(valid_from AS DATE) AS BIGINT)
    END                                                      AS version_duration_days
FROM dim_accounts
ORDER BY account_id, valid_from;
```

### Column Descriptions

| Column | Type | Description |
|--------|------|-------------|
| `account_sk` | `BIGINT` | Surrogate key (from `dim_accounts`). |
| `account_id` | `VARCHAR` | Natural key (from `dim_accounts`). |
| `primary_currency` | `VARCHAR` | Mode currency for this version (from `dim_accounts`). |
| `primary_category` | `VARCHAR` | Mode category for this version (from `dim_accounts`). |
| `transaction_count` | `BIGINT` | Transaction count for this version (from `dim_accounts`). |
| `total_spend` | `DOUBLE` | Total spend for this version (from `dim_accounts`). |
| `first_seen` | `DATE` | Earliest transaction date for this version (from `dim_accounts`). |
| `last_seen` | `DATE` | Latest transaction date for this version (from `dim_accounts`). |
| `valid_from` | `TIMESTAMPTZ` | Version start timestamp (from `dim_accounts`). |
| `valid_to` | `TIMESTAMPTZ` | Version end timestamp (from `dim_accounts`). Sentinel `9999-12-31` for current rows. |
| `is_current` | `BOOLEAN` | Whether this is the active version (from `dim_accounts`). |
| `version_number` | `BIGINT` | Chronological version counter per account. Version 1 is the first version ever created for that account. Computed via `ROW_NUMBER() OVER (PARTITION BY account_id ORDER BY valid_from ASC)`. |
| `version_duration_days` | `BIGINT` | Number of days this version was or has been active. For expired rows: `valid_to::DATE - valid_from::DATE`. For the current row: `CURRENT_DATE - valid_from::DATE` (avoids the misleading large number that would result from subtracting against the sentinel date). |

## Relationships

```
transactions (PK: transaction_id)
    │  account_id VARCHAR
    │
    │  N:1 (many transactions per account)
    ▼
dim_accounts (PK: account_sk, NK: account_id)
    │  account_id is the join key
    │  Attributes derived by aggregating transactions WHERE account_id = ?
    │
    │  1:N (one current row + zero or more expired rows per account_id)
    ▼
mart__account_history (VIEW over dim_accounts)
    Adds version_number and version_duration_days; no new storage.

stg_transactions (VIEW over transactions)
    Used as the source for attribute derivation queries during the dimension
    build (same data as transactions, with column aliases).
```

### Join Patterns

**Fact-to-dimension join** (point-in-time, current attributes only):

```sql
SELECT t.transaction_id, t.amount, d.primary_category
FROM transactions t
JOIN dim_accounts d
  ON t.account_id = d.account_id
 AND d.is_current = true;
```

**Fact-to-dimension join** (as-of historical, for time-travel analysis):

```sql
SELECT t.transaction_id, t.amount, d.primary_category
FROM transactions t
JOIN dim_accounts d
  ON t.account_id = d.account_id
 AND t.timestamp >= d.valid_from
 AND t.timestamp <  d.valid_to;
```

**Note on referential integrity**: DuckDB does not enforce FOREIGN KEY
constraints at DML time. The application layer maintains consistency by always
deriving `dim_accounts` from the `transactions` table — every `account_id` in
`dim_accounts` is guaranteed to exist in `transactions` because the dimension
build queries `transactions` as its source.

## Validation Rules

The following integrity constraints must hold after every successful pipeline run.
These are enforced by the data quality check module (FR-008), not by DDL
constraints.

### QC-001: Exactly one current row per account

```sql
-- Must return zero rows (no violations)
SELECT account_id, COUNT(*) AS current_row_count
FROM dim_accounts
WHERE is_current = true
GROUP BY account_id
HAVING COUNT(*) <> 1;
```

**Invariant**: For every `account_id` that appears in `dim_accounts`, exactly
one row has `is_current = true`. An account that has never been processed will
have no rows; an account that has been processed has exactly one current row.

### QC-002: No overlapping validity ranges per account

```sql
-- Must return zero rows (no overlapping versions)
SELECT a.account_id, a.account_sk, b.account_sk
FROM dim_accounts a
JOIN dim_accounts b
  ON a.account_id = b.account_id
 AND a.account_sk <> b.account_sk
 AND a.valid_from  <  b.valid_to
 AND a.valid_to    >  b.valid_from;
```

**Invariant**: For any given `account_id`, the validity windows of all version
rows are non-overlapping. The `valid_to` of version N equals the `valid_from`
of version N+1.

### QC-003: No null surrogate keys

```sql
-- Must return zero rows
SELECT account_sk FROM dim_accounts WHERE account_sk IS NULL;
```

**Invariant**: Every row in `dim_accounts` has a non-null `account_sk`. This
is guaranteed by the sequence default but verified explicitly.

### QC-004: Current rows use the far-future sentinel

```sql
-- Must return zero rows (current rows must use the sentinel)
SELECT account_sk, account_id, valid_to
FROM dim_accounts
WHERE is_current = true
  AND valid_to <> '9999-12-31 23:59:59+00'::TIMESTAMPTZ;
```

**Invariant**: All rows marked `is_current = true` have `valid_to` set to the
sentinel value `9999-12-31T23:59:59+00:00`. This prevents queries from
accidentally filtering out current rows when using date-range joins.

### QC-005: Contiguous validity windows per account

```sql
-- For every account, valid_to of version N must equal valid_from of version N+1
-- Must return zero rows (no gaps in the timeline)
WITH ordered AS (
    SELECT
        account_id,
        valid_from,
        valid_to,
        LEAD(valid_from) OVER (
            PARTITION BY account_id ORDER BY valid_from ASC
        ) AS next_valid_from
    FROM dim_accounts
)
SELECT account_id, valid_to, next_valid_from
FROM ordered
WHERE next_valid_from IS NOT NULL
  AND valid_to <> next_valid_from;
```

**Invariant**: There are no gaps between consecutive versions of the same
account. The transition from one version to the next is instantaneous at the
pipeline run timestamp.

## State Transitions

### SCD2 Row Lifecycle

```
                          pipeline run N
                         ┌─────────────────────────────────────────────┐
                         │                                             │
 [account_id not in      │   INSERT new row                           │
  dim_accounts]          │   account_sk  = nextval(seq)               │
         ────────────────►   valid_from  = run_timestamp              │
                         │   valid_to    = 9999-12-31T23:59:59+00     │
                         │   is_current  = true                       │
                         │                                             │
                         │              │  NEW → CURRENT              │
                         │              ▼                              │
                         │   ┌─────────────────────┐                  │
                         │   │  is_current = true  │                  │
                         │   │  row_hash   = H     │                  │
                         │   └─────────────────────┘                  │
                         │         │          │                       │
                         │   hash  │          │ hash                  │
                         │ unchanged│          │ changed               │
                         │         │          │                       │
                         │         ▼          ▼                       │
                         │   UNCHANGED    EXPIRE old row              │
                         │   (no writes)  UPDATE SET                  │
                         │                  is_current = false        │
                         │                  valid_to = run_timestamp  │
                         │                                             │
                         │                then INSERT new row         │
                         │                  valid_from = run_timestamp│
                         │                  valid_to = sentinel       │
                         │                  is_current = true         │
                         │                                             │
                         │            CURRENT → EXPIRED + NEW CURRENT │
                         └─────────────────────────────────────────────┘
```

### State Summary

| State | `is_current` | `valid_to` | Description |
|-------|-------------|------------|-------------|
| **NEW** | `true` | `9999-12-31T23:59:59+00` | Account appears for the first time. One row inserted. |
| **CURRENT** | `true` | `9999-12-31T23:59:59+00` | Active version. No change detected since last run. Row untouched. |
| **EXPIRED** | `false` | run timestamp | A prior current row whose attributes changed. Closed in-place by the SCD merge; never deleted. |
| **UNCHANGED** | `true` | `9999-12-31T23:59:59+00` | Existing current row, hash matches recomputed profile. No action taken. |

### Two-Statement SCD Merge Pattern

The dimension build executes exactly two SQL statements inside a single
atomic transaction (FR-011):

**Statement A — Expire changed current rows**:

```sql
UPDATE dim_accounts
SET
    is_current = false,
    valid_to   = ?::TIMESTAMPTZ   -- run_timestamp
WHERE is_current = true
  AND account_id IN (
      SELECT account_id
      FROM   <newly_computed_profiles>
      WHERE  row_hash <> dim_accounts.row_hash
  );
```

**Statement B — Insert new current rows** (for new accounts and changed accounts):

```sql
INSERT INTO dim_accounts (
    account_id, primary_currency, primary_category,
    transaction_count, total_spend, first_seen, last_seen,
    row_hash, valid_from, valid_to, is_current
)
SELECT
    account_id, primary_currency, primary_category,
    transaction_count, total_spend, first_seen, last_seen,
    row_hash,
    ?::TIMESTAMPTZ AS valid_from,   -- run_timestamp
    '9999-12-31 23:59:59+00'::TIMESTAMPTZ AS valid_to,
    true           AS is_current
FROM <newly_computed_profiles> AS p
WHERE NOT EXISTS (
    SELECT 1
    FROM   dim_accounts d
    WHERE  d.account_id  = p.account_id
      AND  d.is_current  = true
      AND  d.row_hash    = p.row_hash
);
```

**Transaction wrapper** (FR-011, R-004):

```python
conn.begin()
try:
    conn.execute(EXPIRE_SQL, [run_timestamp])
    conn.execute(INSERT_SQL, [run_timestamp])
except Exception:
    conn.rollback()
    raise
else:
    conn.commit()
```

### Attribute Derivation Query

Account attributes are computed from the **full** transaction history before
each SCD merge. The `row_hash` is computed over the pipe-delimited concatenation
of the six tracked attributes (R-002, R-003):

```sql
WITH currency_counts AS (
    SELECT account_id, currency, COUNT(*) AS cnt
    FROM   stg_transactions
    GROUP  BY account_id, currency
),
primary_currency AS (
    SELECT account_id, currency AS primary_currency
    FROM   currency_counts
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY account_id
        ORDER BY cnt DESC, currency ASC
    ) = 1
),
category_counts AS (
    SELECT account_id, category, COUNT(*) AS cnt
    FROM   stg_transactions
    GROUP  BY account_id, category
),
primary_category AS (
    SELECT account_id, category AS primary_category
    FROM   category_counts
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY account_id
        ORDER BY cnt DESC, category ASC
    ) = 1
),
account_stats AS (
    SELECT
        account_id,
        COUNT(*)            AS transaction_count,
        SUM(amount)         AS total_spend,
        MIN(transaction_date) AS first_seen,
        MAX(transaction_date) AS last_seen
    FROM stg_transactions
    GROUP BY account_id
)
SELECT
    s.account_id,
    c.primary_currency,
    g.primary_category,
    s.transaction_count,
    s.total_spend,
    s.first_seen,
    s.last_seen,
    md5(
        COALESCE(c.primary_currency, '') || '|' ||
        COALESCE(g.primary_category, '') || '|' ||
        COALESCE(CAST(s.transaction_count AS VARCHAR), '') || '|' ||
        COALESCE(CAST(s.total_spend AS VARCHAR), '')      || '|' ||
        COALESCE(CAST(s.first_seen AS VARCHAR), '')       || '|' ||
        COALESCE(CAST(s.last_seen AS VARCHAR), '')
    ) AS row_hash
FROM   account_stats  s
JOIN   primary_currency c USING (account_id)
JOIN   primary_category g USING (account_id);
```

## Migration Considerations

`dim_accounts` and `dim_accounts_sk_seq` are **entirely new objects**. No
existing table is altered. No data migration from existing schema is required.

### Required Changes to `loader.py`

The `create_tables()` function in `src/ingestion/loader.py` must be extended
to create the sequence and dimension table on pipeline startup. Add the
following DDL constants and include them in the `create_tables()` call:

```python
_DIM_ACCOUNTS_SK_SEQ_DDL: str = """
CREATE SEQUENCE IF NOT EXISTS dim_accounts_sk_seq START 1;
"""

_DIM_ACCOUNTS_DDL: str = """
CREATE TABLE IF NOT EXISTS dim_accounts (
    account_sk          BIGINT      PRIMARY KEY
                                    DEFAULT nextval('dim_accounts_sk_seq'),
    account_id          VARCHAR     NOT NULL,
    primary_currency    VARCHAR     NOT NULL,
    primary_category    VARCHAR     NOT NULL,
    transaction_count   BIGINT      NOT NULL,
    total_spend         DOUBLE      NOT NULL,
    first_seen          DATE        NOT NULL,
    last_seen           DATE        NOT NULL,
    row_hash            VARCHAR     NOT NULL,
    valid_from          TIMESTAMPTZ NOT NULL,
    valid_to            TIMESTAMPTZ NOT NULL,
    is_current          BOOLEAN     NOT NULL
);
"""
```

In `create_tables()`, add before the existing `logger.info("Warehouse tables ready")` line:

```python
conn.execute(_DIM_ACCOUNTS_SK_SEQ_DDL)
conn.execute(_DIM_ACCOUNTS_DDL)
```

Both DDL statements are idempotent (`CREATE ... IF NOT EXISTS`) and safe to
call on every pipeline run against both fresh and existing databases.

### New Module: `src/dimensions/`

The SCD merge logic must be implemented as a dedicated Python module separate
from `loader.py` and the SQL transform runner (FR-012):

```
src/
  dimensions/
    __init__.py
    dim_accounts.py    # build_dim_accounts(conn, run_timestamp) → DimBuildResult
```

The `dim_accounts.py` module contains the attribute derivation query, the
two-statement SCD merge, and the atomic transaction wrapper. It is invoked
by the pipeline orchestrator after transforms and before quality checks (FR-009).

### Pipeline DAG Integration

The dimension build step is inserted between the SQL transform runner and the
data quality checks:

```
ingestion → transforms → [NEW] dimension build → quality checks
```

The dimension build result (accounts processed, new versions created, unchanged
accounts) is recorded in pipeline run tracking (FR-010).
