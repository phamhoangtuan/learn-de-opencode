# Implementation Plan: SCD Type 2 — Accounts Dimension

**Branch**: `009-scd2-accounts` | **Date**: 2026-02-26 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/009-scd2-accounts/spec.md`

## Summary

Build a `dim_accounts` SCD Type 2 dimension table that derives behavioral account profiles (primary currency, primary category, transaction count, total spend, first/last seen dates) from the transactions table and tracks attribute changes over time using valid_from/valid_to/is_current versioning. Implemented as a dedicated Python module (`src/dimensions/`) using DuckDB SQL with atomic transactions, hooked into the existing pipeline orchestrator after transforms and before quality checks.

## Technical Context

**Language/Version**: Python 3.11+
**Primary Dependencies**: duckdb (existing), polars (existing — for type hints; SCD merge is pure SQL)
**Storage**: DuckDB embedded warehouse (`data/warehouse/transactions.duckdb`)
**Testing**: pytest (existing — unit + integration + quality tiers)
**Target Platform**: Linux/macOS single-node (same as existing pipeline)
**Performance Goals**: Dimension build < 5 seconds for up to 100,000 transactions (SC-005)
**Constraints**: Full atomicity required — no partial writes permitted (FR-011); all SCD changes in one `conn.begin()` transaction
**Scale/Scope**: Expected < 100K unique accounts; dim_accounts rows grow by at most N new versions per run

## Phase 0: Research ✅

See [research.md](research.md) for full findings. Summary:

| # | Topic | Decision |
|---|-------|----------|
| R-001 | Surrogate key | DuckDB `SEQUENCE` → `BIGINT` via `nextval()` at INSERT time |
| R-002 | Change detection | MD5 row-hash + two-statement MERGE (expire → insert) |
| R-003 | Mode tie-breaking | `ROW_NUMBER() OVER (ORDER BY cnt DESC, value ASC) QUALIFY rn = 1` |
| R-004 | Atomicity | `conn.begin()` / `conn.commit()` / `conn.rollback()` in `try/except/else` |

## Phase 1: Design ✅

### Data Model

See [data-model.md](data-model.md). Key entities:

**`dim_accounts_sk_seq`** — DuckDB sequence, START 1
**`dim_accounts`** — 13-column SCD2 table:
- `account_sk BIGINT PK` — surrogate key (sequence)
- `account_id VARCHAR NOT NULL` — natural key
- `primary_currency VARCHAR` — mode of currency, alpha tie-break
- `primary_category VARCHAR` — mode of category, alpha tie-break
- `transaction_count BIGINT` — COUNT(*) from full history
- `total_spend DOUBLE` — SUM(amount) from full history
- `first_seen DATE` — MIN(transaction_date)
- `last_seen DATE` — MAX(transaction_date)
- `row_hash VARCHAR NOT NULL` — MD5 fingerprint of the 6 tracked attributes
- `valid_from TIMESTAMPTZ NOT NULL` — when this version became active
- `valid_to TIMESTAMPTZ` — NULL = current row; set to run_date - 1 day when expired
- `is_current BOOLEAN NOT NULL DEFAULT TRUE`
- `run_id VARCHAR NOT NULL` — lineage: pipeline run that created this version (constitution §II)

**`mart__account_history`** — SQL VIEW adding `version_number` (ROW_NUMBER per account) and `version_duration_days` (CASE WHEN is_current THEN today - valid_from ELSE valid_to::DATE - valid_from::DATE END).

### Contracts

See [contracts/dim_builder.md](contracts/dim_builder.md) and [contracts/quality_checks.md](contracts/quality_checks.md).

**Module interface** (`src/dimensions/dim_accounts.py`):
- `create_dim_tables(conn)` — idempotent DDL
- `build_dim_accounts(conn, run_id, run_date) -> DimBuildResult` — atomic SCD2 merge
- `DimBuildResult(accounts_processed, new_versions, unchanged, elapsed_seconds)`
- `DimBuildError` — raised on any failure; rollback guaranteed

**Quality checks** (3 × critical severity):
- `check__dim_accounts_single_current` — exactly one `is_current=TRUE` per account_id
- `check__dim_accounts_no_overlapping_ranges` — no overlapping valid windows
- `check__dim_accounts_no_null_sk` — no null surrogate keys

### Pipeline Integration

The dimension build inserts as a new step in `run_pipeline.py` between `transforms` and `checks`:

```
generate → ingest → transforms → dim_build → checks → dashboard
```

The step calls `build_dim_accounts()` and writes `DimBuildResult` stats to a new `dim_build_runs` metadata table (T032).
